Notas sobre la Google Test Automation Conference 2013 #GTAC2013

Escrito por kinisoftware en la bitácora Kinisoftware, artículo original
Publicado el lunes, 04 de noviembre de 2013 a las 08:28 UTC
Etiquetado como uncategorized

Aunque esta conferencia no es reciente, tuvo lugar en Abril de este año (2013), acabo de terminar de ver los vídeos que más me interesaban y tomar notas sobre ellos. Ya que me ha tomado un tiempo y creo que me resultará útil he pensado que igual podría ser útil para más gente y aquí comparto las cosas que apunté

Sobre todo he visto las charlas relacionadas con Android. Molan mucho todas en general porque cuentan como hacen empresas como Google o Facebook sus procesos reales de testing sobre aplicaciones móviles. Se pueden sacar muchas ideas

https://developers.google.com/google-test-automation-conference/2013/presentations

  • Android UI Automator

    • http://developer.android.com/tools/testing/testing_ui.html

    • No customize views yet (as WebView)

    • Based on accessibility properties

    • Blackbox from app point of view

    • API 16 or higher

    • JUnit notation (Test runner and test case base classes)

    • UI Automator View Tool (inspect AUT via accessibility framework)

  • Espresso

    • Screencast example https://www.youtube.com/watch?feature=player_embedded&v=qtKx1WxK7cw

    • UI Testing

    • https://code.google.com/p/android-test-kit/wiki/Espresso

    • Easy

      • Find (onView); Do (perform); Check (check)

      • No “wait” or boilerplate

      • Extensible API (use Hamcrest)

    • Reliable

      • UI Events; Async Task => Espresso does

      • Fast

    • Durable

      • Several ways to find a resource in the UI

    • Android

      • UI hermetic tests with Robotium (well, Goobotium => will be integrated to Espresso… eventually)

      • Fake G+ server; Fake services (as fake login server and so on)

      • Monkey tests

    • iOS

      • Unit test suite based on GTMSenTestcase

      • Hermetic UI tests (KIF)

      • Mock server

    • G+ backend

      • Replay test that check response from server with simulate mobile clients requests (Golden requests => Golden responses; Fake services)

  • Mobile Automation in Continuos Delivery Pipeline

    • CD = CI + C Test Automation + Continuous Deployment

    • CD Pipeline

      • Pre-commit => Unit test

        • Trunk => Pre-promote test (JS Unit Test Jasmine; Trunk Smoke Test)

          • Promote => Post-promote Testing (Acceptant test)

            • Trunk Deploy Stubbed => iPhone; Android; Chrome

      • Fix forward in 10 or revert

      • Build Coach to take care of fix the break build (rotate responsibility)

      • Mobile Web

        • Jasmine (JavaScript)

        • Cucumber (UI/API) for BDD

        • Selenium-Webdriver

        • TestNG (unit) fot TDD (great for parallel tests) http://testng.org

        • EasyMock

      • Mobile Apps

        • iOS => Frank; Cucumber

  • WebDriver for Chrome

    • ChromeDriver follows the W3C WebDriver working draft

    • Open source and maintained by members of the Chromium project

    • https://sites.google.com/a/chromium.org/chromedriver/

  • Appium

    • http://appium.io/

    • Cross-platform solution for native and hybrid mobile automation

      • Write one test and pass in both platforms (Android & iOS)

      • Support web testing (Mobile Safari & Chrome Android)

      • Simulators/Emulators and real devices

    • Philosophy:

      • Test the same app you submit to the marketplace

        • Test the binary

      • Write your tests in any language, using any fw

        • Agnostic and general

      • Use a standard automation specification API

      • Build a large open-source community

        • User support

        • Contribution support

    • Selenium WebDriver base (http://selendroid.io/)

    • Architecture:

      • an HTTP Server that creates and handles WebDriver sessions

      • starts a test case on the device (listens for proxies commands)

        • iOS => proxies commands to a UIAtomation script running in Instruments (bootstrap.js)

        • Android => proxies commands to a UiAutomator test case running on the device (bootstrap.jar)

    • Opens the door to cross-platform mobile testing => one test, two mobile platforms

    • Demo (code):

      • Define capabilities for both platforms (configure the app to lunch)

      • Define differences (selectors, text and so on)

    • Appium Inspector (Record and play)

    • Limitation => Scaling up for use in CI

    • Android => Native to WebViews and back is okay; iOS is not possible yet

  • How Facebook tests Facebook on Android

    • No Android Team

      • Each feature is owned by a team and they are in charge of release that feature in every platform (Web, Android, iOS, etc)

    • Build

      • Git

      • Two branches (no presubmit)

        • Master => latest version of the code, linear history

        • Stable => the last time they are sure that unit test were passing

      • Code organization

        • Single Tree => three/four libraries: Java, Java test (they do not mix test and Java code), third-party libraries and resources.

      • Working way:

        • Create a branch from stable

        • Hacking/Commit/Hacking/Commit

        • Rebase from origin as many as possible

        • Hacking/Commit/Hacking/Commit

        • Rebase in order to keep a clean and linear history with logical changes on it

      • IntelliJ

      • Dependency Injection

        • OOO => Collaboration

      • Buck (http://facebook.github.io/buck/) => Building code creating by Facebook

    • Test

      • No test department, they just have engineers. Some of them have some passion for testing as another has for security and so on

      • Unit test

        • JUnit

        • EasyMock

        • Roboelectric

      • Larger tests

        • UiAutomator

        • Bully (framework in-house – no open source yet)

        • Roboelectric

  • Evolution from Quality Assurance to Test Engineering

    • In Google

      • Single monolithic code tree with mixed language code

      • Development on one branch (everything is done at head)

      • All builds from source (even small builds)

    • Tools

      • Build system

        • Performance => Parallelize as much as they can

        • Correctness => Make clean is a bug in the system

        • Incrementally => The least amount of work, be able to reuse an output from another developer across the company (same target, same revision; CI use a cache)

        • Ease of use

        • Multi-functional => Different operations over a particular target with just one command

        • Multi-language => All is a target, it does not mind the language

      • Test execution

        • Also happen in parallel

        • A test starts as soon as its build is ready; One test fail, the build process fails

      • Google’s CI

        • Presubmit => Take some code I am editing on my workstation, hand it over to the CI, merge it with the last green build and tell me if I’m going to break the build

        • Code Reviews is also sent to the CI (not just to another programmer) so at the same time you know if your test pass, fail, what failed, logs, incremental code coverage (for the code you just added the tests you wrote just cover a 23%, for example, of the new code) and more

        • Fine grained => They know what tests are affected by a particular change so they just run those tests

        • Flaky tests => nondeterministic.

          • Keeps track of flaky information (reports for flaky tests)

          • Retry tests

          • Extra analysis flaky tests overnight

    • Test engineering

      • Create processes and plans to test a particular product

      • Determine the best way to maximize coverage

      • Design and implement test infrastructure and rapid development and release tools

      • Best practices

      • Hermetic Environments http://googletesting.blogspot.com.es/2012/10/hermetic-servers.html

        • The less external dependencies they are, the better

        • N instances so they can run test in parallel

        • Separe environments

      • Metrics => If you cannot measure it, you cannot understand or control it

  • Breaking the Matrix – Android Testing at Scale (at Google)

    • Why are not we automating? Problems =>

      • Having a place to run tests

      • Any device at your fingertips

      • Making it fun to write tests

    • Having a place to run tests

      • Having a physical lab presents many problems

      • What are the most common bugs? => The most bugs you can catch do not care what phone they are running on so hardware (physical device) is not so important

      • Catching bugs!

        • Replicate (emulators):

          • OS Version

          • Screen size

          • Memory constraints

        • Only a few classes of bugs will slip thru our net.

        • Then we can write targeted tests for our pieces of code that interact with Bluetooth, GPS, battery, CPU and so on. And launch them in real devices.

        • Exploratory tests like real users, not just running scenarios written in a spreadsheet

      • Emulators => Data center => scale and reproducible test environment

    • Any device at your fingertips

      • Emulators are not realistic?

        • It is just another android device => Goldfish, the emulator hardware

        • It emulates real phone hardware

        • System images are almost identical

      • Most of the bugs will not occur in the Device Specific layer of Android System Architecture

      • Create configurations to match devices specifications (RAM, screen resolution, screen density, orientation)

      • Operating System modifications (all images are root by default)

        • Override system properties

        • Test friendly services

        • More logging

      • Launching by command line (device, API and so on)

      • Emulators are sooo slow:

        • Hibernation => Snapshot

        • Get rid of ARM CPU => Running on host CPU (VM Acceleration)

    • Making it fun to write tests

      • Android activity => View (MVC) of Presenter (MVP)?

      • Android Instrumentation (Robotium over android instrumentation)

      • Flaky tests

        • Concurrency

      • Google Android Testing FW => Espresso
