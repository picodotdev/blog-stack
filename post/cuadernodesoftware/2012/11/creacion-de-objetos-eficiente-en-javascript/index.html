Creación de objetos eficiente en JavaScript

Escrito por Guille en la bitácora Cuaderno de software, artículo original
Publicado el viernes, 02 de noviembre de 2012 a las 12:17 UTC
Etiquetado como prototipos, desarrollo-de-software, javascript

There’s more than one way to skin a cat

– Proverbio inglés.

Hay muchas maneras de crear objetos en JavaScript y la que elijamos debe depender de las necesidades particulares de cada problema, pero siempre tenemos que tener en mente la eficiencia del código que escribamos.

En general, queremos que nuestro código fuente se ejecute una sola vez y que prepare en memoria todo lo necesario para generar objetos, uno detrás de otro, como churros y con el mínimo esfuerzo posible. Por ello, la mejor opción consiste en usar una función constructora y añadir aquello que no sea el estado de una nueva instancia en el prototipo de la función constructora:

var Counter = function(offset) {
  this.value = offset || 0;
};
 
Counter.prototype.increase = function() {
  return ++this.value;
};
 
Counter.prototype.decrease = function() {
  return --this.value;
};

Con este código, cada vez que hagamos un new Counter() solo estaremos ejecutando la única línea de nuestro constructor – que inicializa el estado de la nueva instancia – y los métodos se heredarán del prototipo de Counter.

Sin embargo, esta solución nos impide ocultar el estado de las instancias. En algunos casos es deseable y en otros no pero siempre viene bien poder hacer algo “privado”, para que nadie lo pueda sobreescribir o para, sencillamente, no exponer cosas innecesarias y simplificar el API de nuestros objetos.

JavaScript es un lenguaje puñetero, porque la visibilidad de objetos depende del contexto en el que son accedidos. Los contextos en JavaScript son bloques de código entre dos {} y en general, desde uno de ellos, solo tienes acceso a lo que en él se defina y a lo que se defina en otros contextos que contengan al tuyo. Esto hace que el siguiente ejemplo no funcione:

var Counter = function(offset) {
  var value = offset || 0; // value solo es visible en esta función
};
 
Counter.prototype.increase = function() {
  return ++value; // undefined
};
 
Counter.prototype.decrease = function() {
  return --value; // undefined
};

Como value solo es visible en la función constructora (donde la definimos), debemos mover la implementación de los dos métodos a la función constructora:

var Counter = function (offset) {
  var value = offset || 0;
 
  this.increase = function () {
    return ++value;
  };
 
  this.decrease = decrease = function () {
    return --value;
  };
};

Aunque ahora el código funciona y hemos conseguido ocultar el estado de las nuevas instancias que creemos, esta solución tiene dos graves problemas:

  1. Los métodos se implementan en las instancias cada vez que son creadas y esto tiene un impacto negativo en la eficiencia del código
  2. Los métodos se implementan en las instancias, no en su prototipo. Cada instancia tiene sus propios métodos increase() y decrease(), por lo que ocupan más espacio en memoria y limitan nuestra maniobrabilidad para realizar cambios en tiempo de ejecución que afecten a todas las instancias al mismo tiempo. En otras palabras: no estamos usando la orientación a objetos prototípica de manera adecuada.

Como conclusión, tenemos que pensarnos mucho cuándo usar objetos “privados” y cuándo no: ¿podemos hacer lo mismo de otro modo? ¿vale la pena el impacto en la eficiencia?

Bola extra

Si encapsulamos la función constructora en un módulo, nuestro código es más ordenado,evitamos posibles problemas de conflictos de nombre y encima tenemos la posibilidad de manejar alguna “constante” o funcionalidad común no dependiente del estado de las instancias de manera pseudo-privada:

var Counter = (function (undefined) {
 
  // Las funciones increase() y decrease() 
  // solo son accesibles desde el módulo
  function increase(value) {
    return value + 1;
  }
 
  function decrease(value) {
    return value - 1;
  }
 
  var CounterConstructor = function Counter(offset) {
    this.value = offset || 0;
  };
 
  CounterConstructor.prototype.increase = function () {
    this.value = increase(this.value);
    return this.value;
  };
 
  CounterConstructor.prototype.decrease = function () {
    this.value = decrease(this.value);
    return this.value;
  };
 
  return CounterConstructor;
})();
 
// Counter contiene el resultado de la ejecución del 
// Módulo, o sea CounterConstructor
var c = new Counter();

En este ejemplo estamos matando moscas a cañonazos, pero en un caso más complejo nos ayuda a mejorar la organización y encapsulación de nuestro código.

Otro patrón muy común consiste en quitar la sentiencia return y pasarle a la IIFE un objeto que contendrá nuestro constructor:

window.cocotero = window.cocotero || {};
 
(function (ns, undefined) {
 
  // Las funciones increase() y decrease()
  // solo son accesibles desde el módulo
  function increase(value) {
    return value + 1;
  }
 
  function decrease(value) {
    return value - 1;
  }
 
  var CounterConstructor = function Counter(offset) {
    this.value = offset || 0;
  };
 
  CounterConstructor.prototype.increase = function () {
    this.value = increase(this.value);
    return this.value;
  };
 
  CounterConstructor.prototype.decrease = function () {
    this.value = decrease(this.value);
    return this.value;
  };
 
  ns.Counter = CounterConstructor;
})(window.cocotero);
 
// Ahora el constructor está en 
// el namespace 'cocotero'
var c = new cocotero.Counter();

Gracias a esta técnica reducimos la posibilidad de conflictos de nombre aún más gracias al uso de objetos que simularán namespaces en nuestra aplicación.
